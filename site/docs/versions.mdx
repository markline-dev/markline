Markline serves multiple **versions** (v2, v1, вЂ¦) or **locales** (en, es, вЂ¦)
from a single project. Both use the same idea: the **first** entry is the
default and lives at the root; every other entry is served under a **URL
prefix** with its own content folder вЂ” and the header shows a switcher.

## How it works

There's one rule for the whole folder layout: **the default version lives at the
content root; every other version lives in a folder named after its `id`.** A
version folder is just a smaller mirror of the root вЂ” same `docs/` and
`api/openapi.json` inside.

```
content/
в”њв”Ђв”Ђ markline.json          # config + the versions list
в”њв”Ђв”Ђ docs/                  # DEFAULT version docs   в†’ /quickstart
в”‚   в””в”Ђв”Ђ quickstart.mdx
в”њв”Ђв”Ђ api/
в”‚   в””в”Ђв”Ђ openapi.json       # DEFAULT version spec    в†’ /api-reference
в””в”Ђв”Ђ v1/                    # one folder per non-default version id
    в”њв”Ђв”Ђ docs/              # v1 docs                 в†’ /v1/quickstart
    в”‚   в””в”Ђв”Ђ quickstart.mdx
    в””в”Ђв”Ђ api/
        в””в”Ђв”Ђ openapi.json   # v1 spec                 в†’ /api-reference/v1
```

So a version `id` is **both** a folder name and a URL prefix. The one wrinkle to
remember вЂ” docs and the API reference prefix the `id` in *different spots*:

| Content file | Default URL | Version `v1` URL |
| --- | --- | --- |
| `docs/quickstart.mdx` | `/quickstart` | `/v1/quickstart` |
| `api/openapi.json` | `/api-reference` | `/api-reference/v1` |

(Docs put the version first; the API reference keeps everything under
`/api-reference` and puts the version after it вЂ” so all API routes stay in one
place.)

<Note>A project uses **versions or locales**, not both вЂ” they share the
prefix mechanism (if both are set, versions win). The default entry is always
unprefixed, so existing URLs don't change when you add a second version. A
version folder only needs the parts that differ вЂ” drop `api/` and it's
docs-only; drop `docs/` and it's reference-only.</Note>

## Versions

List versions in `markline.json`. The first is the default; each entry is
`{ id, label }` and may carry its own `navigation`.

```jsonc
"versions": [
  { "id": "v2", "label": "v2 (latest)" },   // default вЂ” content in docs/, served at /
  { "id": "v1", "label": "v1" }             // content in v1/docs/, served at /v1/вЂ¦
]
```

<Steps>
  <Step title="Add the version content">
    Create `v1/docs/` and author its pages вЂ” they're independent from the
    default, so v1 can lag or differ freely.
  </Step>
  <Step title="(Optional) give it its own navigation">
    A version entry can set `navigation: { tabs: [...] }` to override the
    sidebar for that version; otherwise it inherits the default navigation.
  </Step>
  <Step title="Ship">
    `markline build` emits every version statically. The header shows a
    **version switcher**; picking one navigates under its prefix.
  </Step>
</Steps>

## API reference per version

Each version can ship its **own OpenAPI spec**. Drop it at
`<id>/api/openapi.json` and Markline serves that version's reference at
`/api-reference/<id>` вЂ” generated exactly like the default, with its own
resource pages, code samples, and interactive explorer.

```
content/
в”њв”Ђв”Ђ api/openapi.json        # default в†’ /api-reference
в””в”Ђв”Ђ v1/
    в””в”Ђв”Ђ api/openapi.json     # v1      в†’ /api-reference/v1
```

- The **version selector** in the API-reference header lists every version and
  switches the spec in place (the default is marked `Default`).
- Within a version, every link вЂ” resource pages, search results, the explorer's
  endpoint switcher вЂ” stays under that version's `/api-reference/<id>` prefix, so
  you never fall back to the default spec mid-navigation.
- Per-resource and per-operation **MDX overlays** are version-scoped too:
  `<id>/api/sections/<tag>.mdx` and `<id>/api/operations/<operationId>.mdx`.
- The pill label follows the spec's `info.version` вЂ” a semver like `1.4.2` shows
  as `v1 В· 1.4.2`; a date like `2025-06-01` shows once, verbatim.

<Note>A version that has no `api/openapi.json` simply has no API reference (its
selector entry won't resolve a spec) вЂ” versions can be docs-only, reference-only,
or both.</Note>

## Localization

Locales work identically вЂ” `i18n.locales`, first is the default (unprefixed),
each other locale under its `id` prefix with content in `<id>/docs/`.

```jsonc

  "locales": [
    { "id": "en", "label": "English" },   // default вЂ” docs/, served at /
    { "id": "es", "label": "EspaГ±ol" }     // es/docs/, served at /es/вЂ¦
  ]
}
```

```

в””в”Ђв”Ђ es/
    в””в”Ђв”Ђ docs/    # Spanish в†’ /es/quickstart
```

The header shows a **language switcher** that jumps to the same page under the
chosen locale prefix.

## Notes

- The **default** entry (first in the list) is never prefixed вЂ” keep your
  canonical content in `docs/`.
- Each variant is fully static and independently navigable; search and the API
  reference work within each.
- Configure these in [Configuration](/configuration); for hosting the built
  output see [Deployment](/deployment).
